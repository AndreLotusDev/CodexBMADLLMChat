# Story 1.2: Basic Shell UI with Navigation

## Status

Draft

## Story

**As a** user,
**I want** to see a basic app shell with navigation between the main screens,
**so that** I can understand the app structure and move between sections.

## Acceptance Criteria

1. App displays a persistent sidebar or top nav with links to: Connection, Schema Browser, Prompt Preview, Settings
2. Each nav item renders a placeholder screen with its name
3. Active nav item is visually highlighted
4. App window has a minimum size of 900×600px and is resizable
5. App uses a dark theme by default

### Architecture: Routing

[Source: architecture/10-frontend-architecture.md#103-routing-architecture]

Use **HashRouter** (not BrowserRouter — Tauri serves files from a local asset server where HTML5 history API routes don't work). Route organization:

```
/ (AppShell — HashRouter)
├── /connection       → ConnectionScreen   (default redirect)
├── /schema           → SchemaBrowserScreen
├── /prompt           → PromptPreviewScreen
└── /settings         → SettingsScreen
```

HashRouter must be placed in `src/main.tsx` wrapping `<App />`.

### Architecture: Component Template Pattern

[Source: architecture/10-frontend-architecture.md#101-component-architecture]

All components must follow the FC pattern:

```typescript
import { FC } from 'react'

interface NavItemProps {
  to: string
  label: string
}

const NavItem: FC<NavItemProps> = ({ to, label }) => { ... }

export default NavItem
```

### Architecture: Dark Mode & Tailwind

[Source: architecture/3-tech-stack.md#31-technology-stack-table]

- **Tailwind CSS 3.x** with `darkMode: 'class'` strategy (NOT `'media'`)
- shadcn/ui pairs with Tailwind and has first-class dark mode support
- Apply `dark` class to `<html>` element — `AppShell.tsx` should do this via `useEffect(() => { document.documentElement.classList.add('dark') }, [])`
- shadcn/ui design tokens (`bg-background`, `text-foreground`, `bg-accent`, `text-accent-foreground`) handle dark/light switching automatically

### Architecture: State Management (for AppShell)

[Source: architecture/10-frontend-architecture.md#102-state-management-architecture]

For this story, only the **router** is needed — no Zustand state is read or written by the shell or placeholder screens. Full `AppState` interface expansion happens in Story 1.3 when connection commands are introduced.

Selector pattern (for future reference, enforce from day 1):
```typescript
// CORRECT — field selector
const activeProfile = useAppStore(s => s.activeProfile)
// WRONG — returns whole store, causes unnecessary re-renders
const { activeProfile } = useAppStore()
```

### Architecture: IPC Gateway (No Changes This Story)

[Source: architecture/17-coding-standards.md#171-critical-fullstack-rules]

`src/commands/index.ts` currently re-exports bare `invoke`. No Tauri commands are invoked in this story (shell + placeholder screens only). **Do not add command wrappers in this story** — that happens in Story 1.3 when backend commands are implemented.

### Architecture: Window Configuration

[Source: architecture/14-deployment-architecture.md — tauri.conf.json]

## Testing

[Source: architecture/16-testing-strategy.md]

**Framework:** Vitest 4.x + jsdom (already configured in `vite.config.ts`)

**Test file location:** `src/__tests__/`

**Setup file:** `src/__tests__/setup.ts` — already mocks Tauri IPC globally; all test files inherit these mocks automatically.

**Test standards for this story:**

```typescript
// AppShell.test.tsx example pattern
import { render, screen } from '@testing-library/react'
import { MemoryRouter } from 'react-router-dom'
import AppShell from '../components/layout/AppShell'

it('renders all 4 nav links', () => {
  render(
    <MemoryRouter initialEntries={['/connection']}>
      <AppShell />
    </MemoryRouter>
  )
  expect(screen.getByText('Connection')).toBeInTheDocument()
  expect(screen.getByText('Schema Browser')).toBeInTheDocument()
  expect(screen.getByText('Prompt Preview')).toBeInTheDocument()
  expect(screen.getByText('Settings')).toBeInTheDocument()
})
```

Install `@testing-library/react` and `@testing-library/jest-dom` if not already present:
```bash
npm install -D @testing-library/react @testing-library/jest-dom
```

Add to `src/__tests__/setup.ts`:
```typescript
import '@testing-library/jest-dom'
```

**Run tests:** `npm run test`

**No backend tests required** — this story touches no Rust business logic (only removes scaffold `greet` command).
