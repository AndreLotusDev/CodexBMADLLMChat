# Story 3.3: Prompt Generation & Copy to Clipboard

## Status

Done

## Story

**As a** user,
**I want** to generate a formatted LLM prompt block from my selected tables and copy it with one click,
**so that** I can immediately paste it into any LLM chat.

## Acceptance Criteria

1. A "Generate Prompt" button produces the prompt from currently selected tables/columns
2. The prompt includes: a preamble ("Here is my database schema:"), DDL for each selected table, and any annotations formatted as comments
3. Prompt Preview screen renders the full generated prompt in a read-only code block
4. A "Copy to Clipboard" button copies the full prompt text with a single click
5. A visual confirmation (toast or button state change) confirms the copy succeeded
6. Regenerating the prompt with a different selection updates the preview immediately

### Utility — Prompt Generator

- [x] Task 2: Create `src/lib/promptGenerator.ts` — pure function (AC: 1, 2, 6)
  - [x] This story builds the prompt assembler in TypeScript. Architecture §6.3 places the Prompt Generator in Rust, but the architecture's IPC signature (`generate_prompt({profileId, selectedTables})` from §5.4) requires a `profileId` that does not exist in this codebase yet — it is introduced by Epic 4.1 ("Save & Load Connection Profiles"). See Dev Notes "Scope: Frontend-Only Prompt Generation" for the full rationale and forward-migration path.
  - [x] Signature:
    ```typescript
    import type { Annotation, PromptBlock, SchemaTree } from '../types'

    export function generatePrompt(
      schemaTree: SchemaTree,
      selectedTables: Set<string>,    // "schema.table" — Story 3.1 selection key format
      selectedColumns: Set<string>,   // "schema.table.column" — Story 3.1 selection key format
      annotations: Map<string, Annotation>, // keyed by buildAnnotationKey(...) — Story 3.2 format
    ): PromptBlock
    ```
  - [x] Behavior — iterate `schemaTree.schemas`, then each `schema.tables`. For every table whose `${schemaName}.${tableName}` is in `selectedTables`, emit a DDL block. Skip tables not in `selectedTables` entirely.
  - [x] **Output format — exact specification:**
    ```
    Here is my database schema:

    {table_block_1}

    {table_block_2}

    ...
    ```
    Where each `table_block` is:
    ```
    {table_annotation_comment_lines_if_any}
    CREATE TABLE {schemaName}.{tableName} (
        {col_line_1},
        {col_line_2},
        ...
        {col_line_N}{,_if_constraints_follow}
        {primary_key_clause_if_any}{,_if_fk_clauses_follow}
        {foreign_key_clause_1}{,_if_more_fks}
        {foreign_key_clause_2}
    );
    ```
  - [x] **Column line format** — for each column whose `${schemaName}.${tableName}.${columnName}` is in `selectedColumns`:
    - Base: `    {columnName} {dataType}` (4-space indent)
    - Append ` NOT NULL` if `isNullable === false`
    - **Column annotation comment**: if `annotations.get(buildAnnotationKey(schemaName, tableName, columnName))` exists, append `  -- {first_line_of_annotation_text}` after the base column declaration. For multi-line annotation text (split on `\n`), emit the first line inline and place additional lines on the line BEFORE the column declaration, each prefixed with `    -- `. (Inline-first-line keeps the common single-line case compact.)
    - **Comma**: every column line ends with `,` EXCEPT the last entry in the block (last column if no constraints follow; otherwise last constraint).
  - [x] **Table annotation comment block** — if `annotations.get(buildAnnotationKey(schemaName, tableName, null))` exists, emit ABOVE the `CREATE TABLE` line. Split text on `\n`; prefix each line with `-- `. Example:
    ```
    -- Stores all registered users
    -- including soft-deleted accounts
    CREATE TABLE public.users (
    ```
  - [x] **PRIMARY KEY clause** — if the table has PK columns AND at least one PK column is in `selectedColumns`, emit:
    ```
        PRIMARY KEY ({pk_col_1}, {pk_col_2})
    ```
    Include ONLY the PK columns that are also in `selectedColumns`. If no PK column is selected, omit the clause entirely. (Indent 4 spaces; comma-separated; no `--` comment on PK lines.)
  - [x] **FOREIGN KEY clauses** — for each `ForeignKey` in `table.foreignKeys`, emit ONLY if the `columnName` is in `selectedColumns`:
    ```
        FOREIGN KEY ({columnName}) REFERENCES {referencedSchema}.{referencedTable}({referencedColumn})
    ```
    Emit one line per FK; each line gets a trailing `,` unless it is the last entry in the block.
  - [x] **Trailing comma discipline** — the last line inside the parens (whether it is a column, the PRIMARY KEY clause, or a FOREIGN KEY clause) gets NO trailing comma. All earlier lines do. This is the standard SQL DDL grammar.
  - [x] **Block separator** — exactly one blank line between table blocks. No trailing blank line at the end of `content` (the final character of `content` should be `;` followed by no newline, OR `;\n` — pick `;\n` for POSIX-friendly text).
  - [x] **Empty selection guard** — if `selectedTables.size === 0`, return `{ content: 'Here is my database schema:\n\n(No tables selected.)\n', tableCount: 0, columnCount: 0, generatedAt: new Date().toISOString() }`. The button that invokes this function should be disabled in this case (Task 5) — the guard exists as defense-in-depth.
  - [x] **Counts**:
    - `tableCount` = number of `selectedTables` entries that were emitted (i.e., that had a corresponding table in the `schemaTree`). A selection key with no matching table in the tree should be silently skipped — defensive against stale selections.
    - `columnCount` = total number of column lines emitted across all tables.
  - [x] **`generatedAt`** = `new Date().toISOString()`.
  - [x] **Iteration order** = the order the tables appear in `schemaTree.schemas[].tables[]`, NOT the insertion order of `selectedTables`. This produces a stable, deterministic output for the same schema regardless of click order — important for AC #6 (regenerating with the same selection produces the same content).

### Components — CopyButton

- [x] Task 4: Create `src/components/prompt/CopyButton.tsx` (AC: 4, 5)
  - [x] **Clipboard mechanism — `navigator.clipboard.writeText()`**, NOT a Tauri IPC command. Tauri v2's WebView2 host runs as a secure context (the `tauri://` and `https://tauri.localhost` schemes both qualify), so the standard `navigator.clipboard` API is available without any plugin. The architecture §5.5 lists `copy_to_clipboard` as a Tauri IPC, but implementing it requires `tauri-plugin-clipboard-manager` which is NOT currently in `Cargo.toml` and `core:clipboard:write-text` is NOT in `src-tauri/capabilities/default.json` (current capabilities: `core:default + opener:default`). Adding the plugin is a real-but-separate piece of work and is deferred — see Dev Notes "Scope: Clipboard via Web API" for the full rationale and forward-migration path.
  - [x] Props:
    ```typescript
    interface CopyButtonProps {
      text: string
    }
    ```
  - [x] Local state machine — three states: `'idle' | 'copied' | 'error'`. Initial: `'idle'`. On click, attempt `await navigator.clipboard.writeText(text)`. On success → `'copied'`. On failure (rejected promise) → `'error'`. In either non-idle state, schedule `setTimeout(() => setState('idle'), 2000)` to auto-revert. Store the timer id in a ref and clear it on unmount and on next click.
  - [x] Render:
    ```tsx
    import { FC, useState, useEffect, useRef } from 'react'
    import { Copy, Check, AlertCircle } from 'lucide-react'
    import { Button } from '@/components/ui/button'

    type State = 'idle' | 'copied' | 'error'

    const CopyButton: FC<CopyButtonProps> = ({ text }) => {
      const [state, setState] = useState<State>('idle')
      const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null)

      useEffect(() => () => {
        if (timerRef.current) clearTimeout(timerRef.current)
      }, [])

      const handleClick = async () => {
        if (timerRef.current) clearTimeout(timerRef.current)
        try {
          await navigator.clipboard.writeText(text)
          setState('copied')
        } catch {
          setState('error')
        }
        timerRef.current = setTimeout(() => setState('idle'), 2000)
      }

      const label = state === 'copied' ? 'Copied!' : state === 'error' ? 'Copy failed' : 'Copy to Clipboard'
      const Icon = state === 'copied' ? Check : state === 'error' ? AlertCircle : Copy

      return (
        <Button onClick={handleClick} variant="default" size="sm" aria-live="polite">
          <Icon size={14} className="mr-2" />
          {label}
        </Button>
      )
    }

    export default CopyButton
    ```
  - [x] All three lucide-react icons (`Copy`, `Check`, `AlertCircle`) are available in the installed version (already used elsewhere: `Search`, `X`, `ChevronRight`, `MessageSquareText`, etc.).
  - [x] `aria-live="polite"` on the button announces the state change to screen readers (`"Copied!"` → `"Copy to Clipboard"` after 2s).

## Dev Notes

### Scope: Frontend-Only Prompt Generation — Rust Migration Deferred

[Source: docs/architecture/5-api-specification-tauri-ipc-commands.md#54-prompt-generation-command, docs/architecture/6-components.md#63-prompt-generator-rust, docs/architecture/11-backend-architecture.md#114-tauri-managed-state]

The architecture places the Prompt Generator in Rust (§6.3) and specifies the IPC signature in §5.4:

```typescript
invoke<PromptBlock>('generate_prompt', {
  profileId: string,
  selectedTables: Array<{ schemaName, tableName, selectedColumns: string[] }>,
}): Promise<PromptBlock>
```

That signature requires `profileId` for two reasons in the architected design:

1. To look up persisted annotations from SQLite (`AnnotationRepository::load_for_profile(profile_id)`).
2. To key the prompt to a saved connection (for telemetry / future caching).

**Neither concern exists in the codebase today as of HEAD:**

- `Cargo.toml` declares only `sqlx` (PostgreSQL). No `rusqlite`, no `keyring`, no `tauri-plugin-clipboard-manager`.
- `src-tauri/src/db.rs` is a TODO stub (per Story 3.2 Dev Notes).
- `src-tauri/src/migrations/` is empty (no `001_initial.sql`).
- `src-tauri/src/repositories/mod.rs` is a stub comment — no `annotation_repository.rs`, no `profile_repository.rs`.
- `src-tauri/src/commands/` contains only `connection.rs`, `schema.rs`, `mod.rs` — no `prompt.rs`, no `clipboard.rs`.
- `src-tauri/src/services/` contains only `connection_manager.rs`, `schema_extractor.rs` — no `prompt_generator.rs`.
- `src/types/index.ts` has no `PromptBlock` interface.
- `src/store/appStore.ts` has no `prompt` or `isGenerating` field, no `setPrompt` action.
- `src/commands/index.ts` exposes only `testConnection`, `connectAndExtractSchema`, `disconnect` — no `generatePrompt`, no `copyToClipboard`.
- `src/screens/PromptPreviewScreen.tsx` is an 8-line stub.
- No `activeProfile` concept in the store — connections in the current build are ad-hoc per session (Story 2.1 stored them outside of a saved profile).

**Implication for Story 3.3**: implementing `generate_prompt` in Rust faithfully to §5.4 would require Epic 4.1 ("Save & Load Connection Profiles") and Epic 4.2 ("Persist Annotations Across Sessions") to land first. Building it ahead of those stories would force an interim signature without `profileId`, then break that signature when Epic 4.1 lands — pointless churn.

The pragmatic alternative — adopted here — is to build the prompt assembler in TypeScript as a pure function. All the inputs (schema tree, selections, annotations) already live in Zustand. The function output (`PromptBlock`) matches architecture §4.5 verbatim, so consuming code does not change when the Rust migration lands.

**Forward-compatibility design choices (so the future Rust migration is mechanical):**

- `PromptBlock` type shape is verbatim from architecture §4.5 (`content, tableCount, columnCount, generatedAt`). No new fields, no renaming.
- The call site in `SchemaBrowserScreen.tsx` is one line: `const block = generatePrompt(schemaTree, selectedTables, selectedColumns, annotations)`. When the Rust IPC lands, it becomes `const block = await commands.generatePrompt({ profileId, selectedTables: ... })` — one file changed, one block changed.
- The TS function reads from Zustand-shaped data, not Rust-shaped data. The Rust version will translate `selectedTables: Set<string>` into the `Array<{schemaName, tableName, selectedColumns: string[]}>` shape inside `src/commands/index.ts` adapter — a thin wrapper, not a redesign.
- Tests target `generatePrompt(schemaTree, ...)` directly. When the IPC migration lands, those tests stay as the spec for the Rust port — the Rust unit tests must produce byte-identical output. (Architecture §16.4 already shows the Rust test shape.)

**Mirrors Story 3.2's pattern.** Story 3.2 deferred SQLite annotation round-trips to Epic 4.2 with the same justification: implementing it ahead of Epic 4.1's `profileId` introduction would create rework. This story extends the pattern one step further.

### Scope: Clipboard via Web API — Tauri Plugin Migration Deferred

[Source: docs/architecture/15-security-and-performance.md#151-security-requirements, docs/architecture/5-api-specification-tauri-ipc-commands.md#55-clipboard-command, docs/architecture/2-high-level-architecture.md#24-high-level-architecture-diagram]

Architecture §5.5 and §2.4 specify a Rust IPC for clipboard:

```typescript
invoke<void>('copy_to_clipboard', { text: string }): Promise<void>
```

Architecture §15.1 lists `core:clipboard:write-text` as a required Tauri capability.

**Codebase reality as of HEAD:**

- `src-tauri/Cargo.toml` does NOT include `tauri-plugin-clipboard-manager`.
- `src-tauri/capabilities/default.json` lists `["core:default", "opener:default"]` only — no clipboard capability.
- `package.json` does NOT include `@tauri-apps/plugin-clipboard-manager`.

**This story uses `navigator.clipboard.writeText()` instead.** Tauri v2's WebView2 host serves the app from a secure context (`tauri://localhost` on Windows, `https://tauri.localhost` on macOS, both secure-context qualified per the W3C Secure Contexts spec), so `navigator.clipboard.writeText` is available without any plugin. The call returns a `Promise<void>` and rejects on permission denial — identical to the architected `commands.copyToClipboard` signature.

**This is a pragmatic deferral, not a deviation from intent.** When a future story adds the Tauri clipboard plugin (likely bundled with Epic 4.x or post-MVP hardening for telemetry/permission auditing), the swap in `CopyButton.tsx` is a one-line change:

```typescript
// This story:
await navigator.clipboard.writeText(text)

// After plugin lands:
await commands.copyToClipboard(text)
```

No call-site changes outside of `CopyButton.tsx`. The state machine, the auto-revert timer, the icon swap, the aria-live region — all unchanged.

**Why defer**: adding `tauri-plugin-clipboard-manager` requires (a) `npm install @tauri-apps/plugin-clipboard-manager --legacy-peer-deps` (per Story 3.1's eslint peer conflict note), (b) `cargo add tauri-plugin-clipboard-manager` (or manual `Cargo.toml` edit, since the user reported in Story 2.2 that VS 2022 Community is not always installed), (c) plugin registration in `src-tauri/src/lib.rs`, (d) capability config in `src-tauri/capabilities/default.json`. None of those add user-visible value beyond what `navigator.clipboard` already delivers in a Tauri WebView2 secure context. The plugin's payoff (Tauri permission auditing, custom MIME types, cross-platform reads) is post-MVP scope.

### Prompt Format — Worked Example

[Source: docs/prd/epic-3-table-selection-prompt-generation.md#story-33, docs/architecture/4-data-models.md#45-promptblock]

Given:

- `schemaTree` with one schema `public` containing tables `users` (cols: `id PK integer NOT NULL`, `email text NOT NULL`, `created_at timestamp`) and `orders` (cols: `id PK integer NOT NULL`, `user_id integer NOT NULL FK→public.users.id`, `total numeric`).
- `selectedTables = { 'public.users', 'public.orders' }`
- `selectedColumns = { 'public.users.id', 'public.users.email', 'public.orders.id', 'public.orders.user_id', 'public.orders.total' }` (i.e., `public.users.created_at` is deselected).
- Annotations:
  - `'public.users.' → { text: 'Stores all registered users' }`
  - `'public.users.email' → { text: 'Used for login and notifications' }`
  - `'public.orders.total' → { text: 'In USD, no tax included\nNegative values are refunds' }`

Expected `content`:

```
Here is my database schema:

-- Stores all registered users
CREATE TABLE public.users (
    id integer NOT NULL,
    email text NOT NULL,  -- Used for login and notifications
    PRIMARY KEY (id)
);

CREATE TABLE public.orders (
    id integer NOT NULL,
    user_id integer NOT NULL,
    -- Negative values are refunds
    total numeric,  -- In USD, no tax included
    PRIMARY KEY (id),
    FOREIGN KEY (user_id) REFERENCES public.users(id)
);
```

And `tableCount === 2`, `columnCount === 5`.

Note in the example:
- `public.users.created_at` is omitted because it is not in `selectedColumns`.
- Table-level annotation for `public.users` is emitted as a `-- ` comment line ABOVE `CREATE TABLE`.
- Single-line column annotation for `email` is inline (`,  -- text`).
- Multi-line column annotation for `total`: first line inline (`,  -- In USD, no tax included`), additional lines emitted on the preceding line(s) as `    -- {line}`.
- `PRIMARY KEY (id)` clause is emitted for both tables because `id` is selected for both.
- `FOREIGN KEY (user_id) REFERENCES public.users(id)` clause emitted because `user_id` is selected.
- Trailing-comma discipline: last line inside each parens has no comma.
- Exactly one blank line between table blocks; final character is `\n` after `;`.

### `PromptBlock` Data Model

[Source: docs/architecture/4-data-models.md#45-promptblock, docs/architecture/10-frontend-architecture.md#102-state-management-architecture]

```typescript
interface PromptBlock {
  content: string         // the full prompt text
  tableCount: number      // count of CREATE TABLE blocks emitted
  columnCount: number     // total column lines emitted across all tables
  generatedAt: string     // ISO 8601 — `new Date().toISOString()`
}
```

Architecture §10.2 prescribes the store layout:

```typescript
prompt: PromptBlock | null
isGenerating: boolean
setPrompt: (prompt: PromptBlock | null) => void
setIsGenerating: (v: boolean) => void
```

Use those names verbatim. `isGenerating` is wired but not animation-driven in this story (prompt generation is synchronous TS — there is no async boundary to gate a spinner against). It is included in the store now for forward-compatibility with the Rust IPC migration, when the async boundary will appear; until then it stays `false` and no component consumes it. No additional UI work is required for `isGenerating` in this story — it is a no-op field that is reset by `clearConnection`.

### Selection Key Format — Reminder

[Source: docs/stories/3.1.story.md, docs/architecture/17-coding-standards.md#171-critical-fullstack-rules]

`generatePrompt` consumes selection keys in the Story 3.1 format:

| Concept | Key format | Built by |
|---------|-----------|----------|
| Selection (table) | `"public.users"` | inline `${schema}.${table}` (Story 3.1) |
| Selection (column) | `"public.users.email"` | inline `${schema}.${table}.${col}` (Story 3.1) |
| Annotation (table) | `"public.users."` (trailing dot) | `buildAnnotationKey(schema, table, null)` (Story 3.2) |
| Annotation (column) | `"public.users.email"` | `buildAnnotationKey(schema, table, col)` (Story 3.2) |

Annotation column key and selection column key happen to be identical in spelling (`public.users.email`) — that is intentional and safe because they live in different containers (`annotations: Map<...>` vs `selectedColumns: Set<...>`).

`generatePrompt` MUST use `buildAnnotationKey()` when looking up annotations (architecture §17.1: "Annotation Key Convention: Always use `buildAnnotationKey()` from `src/lib/utils.ts` — never construct keys inline"). Do not inline `${schemaName}.${tableName}.${columnName ?? ''}`.
