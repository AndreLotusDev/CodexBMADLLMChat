# Story 4.1: Save & Load Connection Profiles

## Status

Draft

## Story

**As a** user,
**I want** to save my PostgreSQL connection details as a named profile,
**so that** I can reconnect to my databases without re-entering credentials each time.

## Acceptance Criteria

1. After a successful connection, the user is prompted to save it as a named profile
2. Saved profiles appear in a dropdown or list on the Connection screen
3. Selecting a saved profile pre-fills all connection fields except the password
4. Password is retrieved securely from Windows Credential Manager — never stored in plain text
5. User can delete a saved profile, which also removes its credentials from Windows Credential Manager
6. At least 10 saved profiles are supported

        let conn = Connection::open(path)
            .map_err(|e| AppError::Internal(format!("Failed to open SQLite at {:?}: {e}", path)))?;

### Credential Store & Error Variants

- [ ] Task 4: Implement `src-tauri/src/credential_store.rs` — WinCred wrapper (AC: 4, 5)
  - [ ] Replace the TODO stub with the architected pattern from `docs/architecture/11-backend-architecture.md#113`:
    ```rust
    use keyring::Entry;
    use crate::errors::AppError;

    pub struct CredentialStore {
        service_name: String,
    }

    impl CredentialStore {
        pub fn new(service_name: impl Into<String>) -> Self {
            Self { service_name: service_name.into() }
        }

        pub fn store(&self, profile_id: &str, password: &str) -> Result<(), AppError> {
            let entry = Entry::new(&self.service_name, profile_id)
                .map_err(|e| AppError::CredentialStoreError(e.to_string()))?;
            entry.set_password(password)
                .map_err(|e| AppError::CredentialStoreError(e.to_string()))
        }

        pub fn retrieve(&self, profile_id: &str) -> Result<String, AppError> {
            let entry = Entry::new(&self.service_name, profile_id)
                .map_err(|e| AppError::CredentialStoreError(e.to_string()))?;
            entry.get_password()
                .map_err(|e| AppError::CredentialNotFound(e.to_string()))
        }

        pub fn delete(&self, profile_id: &str) -> Result<(), AppError> {
            let entry = Entry::new(&self.service_name, profile_id)
                .map_err(|e| AppError::CredentialStoreError(e.to_string()))?;
            match entry.delete_credential() {
                Ok(()) => Ok(()),
                // Treat "not found" as success — idempotent delete (matches the use case: profile delete should not fail if WinCred entry was already cleared)
                Err(keyring::Error::NoEntry) => Ok(()),
                Err(e) => Err(AppError::CredentialStoreError(e.to_string())),
            }
        }
    }
    ```
  - [ ] **Service name = `"schemalift"`** (matches architecture §11.4 `CredentialStore::new("schemalift")` and the productName in `src-tauri/tauri.conf.json`). This becomes the WinCred "service" half of the (service, account) key — the profile UUID is the "account" half.
  - [ ] **Idempotent delete** — `delete()` swallows `NoEntry` errors so that `delete_profile` is safe to call repeatedly without spurious failures. This matters because the profile-delete flow (Task 7) calls `credential_store.delete()` AFTER the SQLite row is deleted; if the WinCred entry was somehow already missing, we don't want to fail the user-facing delete.
  - [ ] **Method is `delete_credential` (keyring 3.x)**, not `delete_password` (keyring 2.x). If the dev finds keyring 2.x is preferable, the method name is `delete_password` instead — verify against the resolved Cargo.lock crate version after `cargo check`.

    pub struct ProfileRepository {
        conn: Arc<Mutex<Connection>>,

    impl ProfileRepository {
        pub fn new(conn: Arc<Mutex<Connection>>) -> Self {
            Self { conn }
        }

        pub fn list(&self) -> Result<Vec<ConnectionProfile>, AppError> {
            let guard = self.conn.lock().map_err(|e| AppError::Internal(format!("DB lock poisoned: {e}")))?;
            let mut stmt = guard.prepare(
                "SELECT id, name, host, port, database, username, created_at
                 FROM connection_profiles
                 ORDER BY name COLLATE NOCASE ASC"
            ).map_err(|e| AppError::Internal(e.to_string()))?;
            let rows = stmt.query_map([], |row| Ok(ConnectionProfile {
                id: row.get(0)?,
                name: row.get(1)?,
                host: row.get(2)?,
                port: row.get::<_, i64>(3)? as u16,
                database: row.get(4)?,
                username: row.get(5)?,
                created_at: row.get(6)?,
            })).map_err(|e| AppError::Internal(e.to_string()))?;
            let mut out = Vec::new();
            for r in rows {
                out.push(r.map_err(|e| AppError::Internal(e.to_string()))?);
            }
            Ok(out)
        }

        pub fn insert(&self, params: &SaveProfileParams) -> Result<ConnectionProfile, AppError> {
            let id = uuid::Uuid::new_v4().to_string();
            let created_at = chrono::Utc::now().to_rfc3339();
            let guard = self.conn.lock().map_err(|e| AppError::Internal(format!("DB lock poisoned: {e}")))?;
            let result = guard.execute(
                "INSERT INTO connection_profiles (id, name, host, port, database, username, created_at)
                 VALUES (?1, ?2, ?3, ?4, ?5, ?6, ?7)",
                params![id, params.name.trim(), params.host.trim(), params.port as i64,
                        params.database.trim(), params.username.trim(), created_at],
            );
            match result {
                Ok(_) => Ok(ConnectionProfile {
                    id, name: params.name.trim().to_string(),
                    host: params.host.trim().to_string(),
                    port: params.port,
                    database: params.database.trim().to_string(),
                    username: params.username.trim().to_string(),
                    created_at,
                }),
                Err(rusqlite::Error::SqliteFailure(sf, _)) if sf.code == rusqlite::ErrorCode::ConstraintViolation => {
                    // CHECK or UNIQUE violation. We don't currently have a UNIQUE(name) constraint — but if a future migration
                    // adds one, this will surface cleanly. The CHECK constraint violations (name_not_empty, port_range) are
                    // also routed here. We bucket all into DuplicateProfileName for now since the only practical violation
                    // a user would hit is empty-name (form validation should prevent it) or duplicate-name. Refine if needed.
                    Err(AppError::DuplicateProfileName)
                }
                Err(e) => Err(AppError::Internal(e.to_string())),
            }
        }

        pub fn delete(&self, profile_id: &str) -> Result<(), AppError> {
            let guard = self.conn.lock().map_err(|e| AppError::Internal(format!("DB lock poisoned: {e}")))?;
            let rows = guard.execute(
                "DELETE FROM connection_profiles WHERE id = ?1",
                params![profile_id],
            ).map_err(|e| AppError::Internal(e.to_string()))?;
            if rows == 0 { return Err(AppError::ProfileNotFound); }
            Ok(())
        }

        pub fn find(&self, profile_id: &str) -> Result<ConnectionProfile, AppError> {
            let guard = self.conn.lock().map_err(|e| AppError::Internal(format!("DB lock poisoned: {e}")))?;
            guard.query_row(
                "SELECT id, name, host, port, database, username, created_at
                 FROM connection_profiles WHERE id = ?1",
                params![profile_id],
                |row| Ok(ConnectionProfile {
                    id: row.get(0)?,
                    name: row.get(1)?,
                    host: row.get(2)?,
                    port: row.get::<_, i64>(3)? as u16,
                    database: row.get(4)?,
                    username: row.get(5)?,
                    created_at: row.get(6)?,
                }),
            ).map_err(|e| match e {
                rusqlite::Error::QueryReturnedNoRows => AppError::ProfileNotFound,
                e => AppError::Internal(e.to_string()),
            })
        }
    }
    ```
  - [ ] **`std::sync::Mutex<Connection>`** (sync) is used, not `tokio::sync::Mutex`. rusqlite calls are synchronous and very fast for the single-row operations in this story; wrapping in `spawn_blocking` would add complexity without measurable benefit at MVP scale (single user, < 50 profiles). The architecture §17.3 shows `spawn_blocking` for "rusqlite in async context" — that pattern is correct for the production path; for MVP this story uses brief `std::sync::Mutex` locks inside async fns. **Cost**: each command call holds the Mutex for sub-millisecond. **Risk**: if the DB grows or queries get slow, the tokio worker thread is blocked. **Mitigation deferred to post-MVP**, documented in Dev Notes "Sync Mutex Trade-off".
  - [ ] **NAME field trimming** — `params.name.trim()` is applied at write time. The CHECK constraint `length(trim(name)) > 0` enforces non-empty at DB level too, so the dual-protection covers UI/IPC bugs.
  - [ ] **Sort order** — list returns profiles ordered by `name COLLATE NOCASE ASC`. Matches the UX expectation that the dropdown is alphabetic, case-insensitive.

    use crate::errors::AppError;
    use crate::models::{ConnectionProfile, SaveProfileParams};
    use crate::AppState;

    #[tauri::command]
    pub async fn list_profiles(
        state: State<'_, AppState>,
    ) -> Result<Vec<ConnectionProfile>, AppError> {
        state.profile_repo.list()
    }

    #[tauri::command]
    pub async fn save_profile(
        name: String, host: String, port: u16, database: String,
        username: String, password: String,
        state: State<'_, AppState>,
    ) -> Result<ConnectionProfile, AppError> {
        let params = SaveProfileParams { name, host, port, database, username, password };
        let profile = state.profile_repo.insert(&params)?;
        if let Err(e) = state.credential_store.store(&profile.id, &params.password) {
            // Compensation: if WinCred write fails, roll back the SQLite row so we don't leave an orphan profile with no credentials.
            let _ = state.profile_repo.delete(&profile.id);
            return Err(e);
        }
        drop(params); // explicit drop hint per architecture §17.3 "Drop sensitive strings explicitly"
        Ok(profile)
    }

    #[tauri::command]
    pub async fn delete_profile(
        profile_id: String,
        state: State<'_, AppState>,
    ) -> Result<(), AppError> {
        state.profile_repo.delete(&profile_id)?;
        state.credential_store.delete(&profile_id)?;
        Ok(())
    }
    ```

  - [ ] **Delete ordering** — SQLite first (because cascade-delete removes annotations atomically), then WinCred (which is idempotent on `NoEntry`). If WinCred fails after SQLite delete succeeded, we surface the error to the user; the profile is gone from SQLite but the WinCred entry leaks. Trade-off: WinCred failure is rare (only on OS-level credential vault issues); the alternative ordering (WinCred first) would leave a phantom profile in SQLite on WinCred success + SQLite failure — worse. Documented trade-off, acceptable for MVP.
  - [ ] **`drop(params)`** — per architecture §17.3 ("Drop sensitive strings explicitly"). The `password` field is inside `params`; dropping it makes the secret-clearing intent explicit.

## Dev Notes

### Scope Summary

This story introduces the entire local-persistence stack for the first time: SQLite via `rusqlite`, the migration runner, the Windows Credential Manager wrapper, the profile repository, and three new IPC commands (`list_profiles`, `save_profile`, `delete_profile`). ### Architectural Deviation — Saved-Profile Connect

[Source: docs/architecture/5-api-specification-tauri-ipc-commands.md#51-connection-commands, current `src-tauri/src/commands/schema.rs`]

The architecture's canonical "connect by profile" signature (§5.1) is:

```typescript
invoke<SchemaTree>('connect_and_extract_schema', { profileId: string }): Promise<SchemaTree>
```

The current code has `connect_and_extract_schema(host, port, database, username, password)` from Story 2.2 — a raw-params signature with no profile awareness (because no profile concept existed at the time).

**This story does NOT refactor the existing handler.** Two reasons:

1. **Backwards-compatibility with the manual-entry path.** AC1 of this story ("After a successful connection, the user is prompted to save it as a named profile") implies that the first connection — before any save happens — uses raw credentials. The existing `connect_and_extract_schema(raw params)` is the right tool for that flow.
2. **Story 2.2 has been signed off as Done.** Refactoring its IPC signature would silently break the SchemaBrowserScreen test fixtures and the existing `ConnectionForm.tsx` wiring without delivering new user value.

**This story adds a sibling command `connect_with_saved_profile({profileId})`** that delivers the architected behavior for the saved-profile path. The Rust handler looks up the profile + WinCred password, builds a `TestConnectionParams`, and delegates to the existing `ConnectionManager::connect()` + `SchemaExtractor::extract()` — zero changes to those services.

**Forward migration to architecture-compliance**: in a post-MVP story (likely 4.4 or later), the raw-params `connect_and_extract_schema` can be deprecated. The migration path: (a) make profile-save the only way to connect (i.e., always auto-save before connecting), (b) remove the raw-params command, (c) rename `connect_with_saved_profile` to `connect_and_extract_schema` to match §5.1. The TS-side `commands.connectWithSavedProfile(profileId)` is the only API call that needs adjustment in that future story.

### Credential Lifecycle — Password Never Touches SQLite

[Source: docs/architecture/15-security-and-performance.md#152-credential-security-model, NFR3 from docs/prd/requirements.md]

Per architecture §15.2 password lifecycle:

```
1. Typed in React form (memory only)
2. Passed to Rust via IPC on save_profile
3. Written to WinCred — never touches SQLite
4. Cleared from Rust stack after WinCred write
5. On connect: WinCred → Rust memory → handshake → dropped
6. NEVER appears in: logs, SQLite, IPC responses, Zustand store
```

**This story's implementation honors all six steps:**

1. ✅ `ConnectionForm.tsx` keeps `password` in React `useState` (local component state, not Zustand) — see existing implementation at `src/components/connection/ConnectionForm.tsx:18`.
2. ✅ `commands.saveProfile({...password})` passes via Tauri IPC.
3. ✅ `commands/profiles.rs::save_profile` calls `credential_store.store(profile_id, password)`. The SQLite `connection_profiles` table has NO `password` column.
4. ✅ `drop(params)` at the end of the handler. (Rust would drop it on function return anyway; the explicit `drop()` documents intent and matches architecture §17.3.)
5. ✅ `commands/connection.rs::connect_with_saved_profile` flow: `credential_store.retrieve()` → moves password into `TestConnectionParams` → `connection_manager.connect(&params)` → `drop(params)`.
6. ✅ `ConnectionProfile` TS interface and the `Vec<ConnectionProfile>` IPC return type have NO password field. The Zustand `AppState` has `savedProfiles: ConnectionProfile[]` and `activeProfile: ConnectionProfile | null` — neither carries a password. The Rust `tracing::*` calls (where present in other modules) never include password fields. **Code review checklist**: grep the diff for `password` and verify every usage is in the credential-store path OR in transient React local state.

**NFR3 ("never plain text")** is satisfied: WinCred stores credentials in the OS vault, which encrypts at rest using the user's Windows logon credentials.

### Sync Mutex Trade-off — Defer `spawn_blocking` to Post-MVP

[Source: docs/architecture/17-coding-standards.md#173-key-rust-patterns]

Architecture §17.3 shows the production pattern for rusqlite in async context:

```rust
let result = tokio::task::spawn_blocking(move || {
    conn.execute(sql, params)
}).await??;
```

This story uses **`std::sync::Mutex<Connection>`** inside async fns directly, NOT `spawn_blocking`. Reasoning:

- **MVP workload**: < 50 profiles, single-row queries, no transactions, sub-millisecond locks.
- **Single-user local app**: no contention; the tokio worker is doing nothing else during these calls.
- **Code clarity**: `spawn_blocking` adds an `Arc::clone` + closure + `.await??` per call site. For a story that's already adding 4 IPC handlers + repo + WinCred + UI, that ceremony is friction without benefit.

**Risk**: if profile/repo operations grow to multi-row queries with non-trivial complexity (joins, larger result sets), the tokio worker could be blocked long enough to delay an unrelated IPC. Mitigation: post-MVP migration to `spawn_blocking` is a mechanical refactor — each repo method gets wrapped, no external API changes. The architecture §17.3 pattern is the eventual target; this story's pragmatic shortcut is documented for future migration.

### Timestamp Format Choice — ISO 8601 via `chrono`

[Source: docs/architecture/4-data-models.md#41-connectionprofile, docs/architecture/9-database-schema.md#91-sqlite-schema-ddl]

The SQLite default `datetime('now')` produces `'2026-05-20 12:34:56'` (space separator, no timezone). The TS interface specifies `createdAt: string` documented as "ISO 8601" — which conventionally means `'2026-05-20T12:34:56Z'` (T separator, Z = UTC). `new Date('2026-05-20 12:34:56')` works in many runtimes but is implementation-defined; `new Date('2026-05-20T12:34:56Z')` is universally parseable.

**This story explicitly inserts a `chrono::Utc::now().to_rfc3339()` value**, overriding the SQLite `DEFAULT`. The DDL keeps the default (for safety / belt-and-braces / migrations from a future DB seed script), but Rust code never relies on it for new inserts.

`chrono::Utc::now().to_rfc3339()` produces `'2026-05-20T12:34:56.123456789+00:00'` (with fractional seconds and offset). This is RFC 3339 (a strict subset of ISO 8601) and parseable by `new Date(...)` in every browser. For display, the dev can use `new Date(profile.createdAt).toLocaleDateString()` — works identically to the pattern in `src/screens/PromptPreviewScreen.tsx:18` (Story 3.3) which uses `toLocaleTimeString()` on a `generatedAt` ISO 8601 string.

### `AppData` Path Resolution — Tauri v2 Manager API

[Source: docs/architecture/9-database-schema.md#92-sqlite-file-location, Tauri v2 docs (https://tauri.app/develop/state-management/)]

Architecture §9.2:

```
Windows: C:\Users\{username}\AppData\Roaming\{bundle_id}\schemalift.db
Resolved at runtime via: tauri::api::path::app_data_dir(&config)
```

That snippet is Tauri v1 API. In **Tauri v2** (this project's framework version), the equivalent is:

```rust
use tauri::Manager;
let app_data_dir = app.path().app_data_dir()?;
```

Where `app` is the `AppHandle` available inside `.setup(|app| { ... })`. The `app_data_dir()` call returns a `Result<PathBuf, tauri::Error>` (NOT the v1 `Option<PathBuf>`).

**Bundle identifier** is set in `src-tauri/tauri.conf.json` to `"com.schemalift.app"`. So the resolved path on Windows is:

```
C:\Users\{username}\AppData\Roaming\com.schemalift.app\schemalift.db
```

`std::fs::create_dir_all(&app_data_dir)` is called before `open_db()` because the directory may not exist on first launch (Tauri doesn't auto-create app data dirs).

### Keyring Crate — Version & API Notes

[Source: docs/architecture/11-backend-architecture.md#113-credential-store, keyring crate docs]

The architecture specifies `keyring` for Windows Credential Manager access (§11.3). This story pins `keyring = "3"`. Key API points:

- `Entry::new(service, account)` — returns `Result<Entry, keyring::Error>`. The `(service, account)` tuple is the WinCred key — `service = "schemalift"`, `account = profile_uuid`.
- `entry.set_password(secret)` — writes to WinCred.
- `entry.get_password()` — reads from WinCred. Returns `Err(keyring::Error::NoEntry)` if not present.
- `entry.delete_credential()` — removes from WinCred. Returns `Err(NoEntry)` if not present. **Note: in keyring 2.x this method is named `delete_password`. Verify the resolved version in `Cargo.lock` after `cargo check` and adjust the method name if needed.**

On Windows targets, `keyring 3.x` uses the WinCred backend by default (`windows-native` feature is on by default). No additional feature flags needed.
