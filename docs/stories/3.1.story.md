# Story 3.1: Table & Column Selection via Checkboxes

## Status

Done

## Story

**As a** user,
**I want** to select tables and columns using checkboxes in the schema tree,
**so that** I can control exactly what context goes into my LLM prompt.

## Acceptance Criteria

1. Every table and column in the schema tree has a checkbox
2. Checking a table automatically checks all its columns
3. Unchecking individual columns is possible after a table is checked (partial selection)
4. A "Select All" / "Deselect All" control is available at the schema level
5. Selected item count is displayed (e.g., "3 tables, 12 columns selected")

### Checkbox Primitive

- [x] Task 3: Create `src/components/ui/checkbox.tsx` — manual primitive (AC: 1)
  - [x] No shadcn CLI (per Story 2.2 Dev Notes — CLI unusable in this project) and no new npm dependency. Use a native `<input type="checkbox">` wrapped to support `indeterminate` via a ref.
  - [x] Props: `{ checked?: boolean; indeterminate?: boolean; disabled?: boolean; onCheckedChange?: (checked: boolean) => void; 'aria-label'?: string; className?: string }`
  - [x] Sketch (do not import `@radix-ui/react-checkbox` — not installed):
    ```typescript
    import { ChangeEvent, FC, useEffect, useRef } from 'react'
    import { cn } from '@/lib/utils'

    interface CheckboxProps {
      checked?: boolean
      indeterminate?: boolean
      disabled?: boolean
      onCheckedChange?: (checked: boolean) => void
      'aria-label'?: string
      className?: string
    }

    export const Checkbox: FC<CheckboxProps> = ({
      checked = false, indeterminate = false, disabled, onCheckedChange,
      className, ...rest
    }) => {
      const ref = useRef<HTMLInputElement>(null)
      useEffect(() => {
        if (ref.current) ref.current.indeterminate = indeterminate && !checked
      }, [indeterminate, checked])

      const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
        onCheckedChange?.(e.target.checked)
      }

      return (
        <input
          ref={ref}
          type="checkbox"
          checked={checked}
          disabled={disabled}
          onChange={handleChange}
          aria-label={rest['aria-label']}
          className={cn(
            'h-4 w-4 shrink-0 rounded border border-border bg-background',
            'text-foreground accent-foreground cursor-pointer',
            'focus:outline-none focus:ring-1 focus:ring-ring',
            'disabled:cursor-not-allowed disabled:opacity-50',
            className,
          )}
        />
      )
    }

    export default Checkbox
    ```
  - [x] `cn` already exists in `src/lib/utils.ts` (used by `button.tsx`, `input.tsx`).
  - [x] `e.stopPropagation()` is NOT needed inside the checkbox itself — the consumers (Tasks 5, 6) wrap the checkbox in a non-button container so the click won't trip the existing `expand/collapse` button.

### Selection State — Data Model

[Source: architecture/4-data-models.md#44-selectionstate-runtime, architecture/10-frontend-architecture.md#102-state-management-architecture]

Architecture §4.4 defines the conceptual `SelectionState`:

```typescript
interface SelectionState {
  tables: Set<string>;    // "schema.table"
  columns: Set<string>;   // "schema.table.column"
}
```

Architecture §10.2 lays out how that conceptual state actually lives in Zustand — as two flat fields on `AppState`:

```typescript
selectedTables: Set<string>
selectedColumns: Set<string>
```

with actions:

```typescript
toggleTable: (schema: string, table: string, columns: PgColumn[]) => void
toggleColumn: (schema: string, table: string, column: string) => void
```

**Use the §10.2 flat-field naming** (`selectedTables`, `selectedColumns`) because that's the concrete Zustand contract. §4.4's `SelectionState` is the abstract model — do not create a nested `selection: SelectionState` field; the architecture's state structure spec wins.

Key formats are exactly:
- Table key: `"schema.table"` (no leading dot, no trailing dot)
- Column key: `"schema.table.column"` (three dot-separated parts)

This matches the prompt-generator's downstream contract (Story 3.3) where the Rust `generate_prompt` command will receive these same string keys.

### Two New Actions Beyond §10.2 — Schema-Level Bulk

The architecture's `AppState` (§10.2) lists `toggleTable` and `toggleColumn` but does not name the schema-level bulk operations needed for AC #4. Add them as natural extensions following the same shape:

```typescript
selectAllInSchema: (schemaName: string, tables: PgTable[]) => void
deselectAllInSchema: (schemaName: string) => void
```

These are new for this story but consistent with the §10.2 action naming convention (camelCase verbs) and with the architecture's intent that "Actions are the only write path" (§10.2 patterns).

### Sync Rule for `selectedTables` ↔ `selectedColumns`

[Source: AC #2 and #3, architecture/4-data-models.md#44-selectionstate-runtime — implied by the prompt-generation contract]

`selectedTables` is the *source of truth* for "this table is included in the prompt." `selectedColumns` is the source of truth for *which* columns of an included table go into the prompt. The two sets must stay coherent:

| Action | `selectedTables` effect | `selectedColumns` effect |
|--------|--------------------------|---------------------------|
| `toggleTable` → check | add `s.t` | add `s.t.c` for every column in `table.columns` |
| `toggleTable` → uncheck | remove `s.t` | remove every `s.t.*` |
| `toggleColumn` → check | ensure `s.t` is present | add `s.t.c` |
| `toggleColumn` → uncheck the last column | remove `s.t` | remove `s.t.c` |
| `toggleColumn` → uncheck a column when others remain | keep `s.t` (indeterminate UI) | remove `s.t.c` |
| `selectAllInSchema` | add `s.t` for every table | add `s.t.c` for every column |
| `deselectAllInSchema` | remove every key with `${schemaName}.` prefix | remove every key with `${schemaName}.` prefix |

The "uncheck the last column → remove table key" rule is the inverse of AC #2 — it keeps the table count in the AC #5 summary truthful (the table is no longer contributing anything).

### Checkbox Tri-State Behavior (AC #3)

[Source: AC #3; architecture/6-components.md#68-schema-browser implies tri-state by mentioning "partial selection"]

A table checkbox has three visual states:

| State | Condition | Visual |
|-------|-----------|--------|
| Unchecked | `selectedColumns` has zero columns for this table | empty box |
| Checked | `selectedTables.has(key)` AND every column is in `selectedColumns` | filled checkmark |
| Indeterminate | at least one but not all columns are in `selectedColumns` | dash / horizontal line |

Native `<input type="checkbox">` supports the `indeterminate` *property* (not an HTML attribute) — set it via a ref + `useEffect` (see Task 3 sketch). React does not have a built-in `indeterminate` prop, which is why the primitive needs the manual ref. Clicking an indeterminate checkbox dispatches `change` with `checked === true` — so the resulting `onCheckedChange(true)` ends up calling `toggleTable`, which currently sees the table key already in `selectedTables` and so will *uncheck* it. **Correction**: the user intent on clicking an indeterminate box is usually "fill it" (check all). To honor that, `toggleTable`'s "table already in `selectedTables`" branch should be the unselect path only when *all* columns are also selected. **For MVP simplicity**, accept the simpler semantic: clicking the table checkbox toggles based on the table-key presence (filled → uncheck; otherwise → check all). Indeterminate users can click the row label/header twice if needed. Document this as a known UX limitation; revisiting it is fine post-MVP and not required for AC pass.
