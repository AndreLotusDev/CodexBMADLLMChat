# Story 2.2: Schema Extraction & Tree View

## Status

Draft

## Story

**As a** user,
**I want** to see the full schema of my connected PostgreSQL database in a tree view,
**so that** I can explore all tables and columns at a glance.

## Acceptance Criteria

1. After a successful connection, the Schema Browser screen loads automatically
2. Tree view displays all schemas → tables → columns hierarchy
3. Each column shows its name, data type, and nullable status
4. Foreign key relationships are indicated on the relevant columns
5. Primary keys and unique constraints are visually distinguished
6. Tree supports expand/collapse at schema and table level
7. Schema extraction for a database with up to 200 tables completes in under 5 seconds

### Schema Extractor — SQL Queries (Batch Strategy)

[Source: architecture/6-components.md#62-schema-extractor-rust, architecture/16-testing-strategy.md]

**Critical**: Do NOT issue per-table queries — that produces 600+ queries for a 200-table DB and violates AC7. Use 5 batch queries, then assemble the tree in Rust memory.

```sql
-- Q1: All non-system schemas
SELECT schema_name
FROM information_schema.schemata
WHERE schema_name NOT IN ('information_schema', 'pg_catalog', 'pg_toast')
  AND schema_name NOT LIKE 'pg_temp_%'
  AND schema_name NOT LIKE 'pg_toast_temp_%'
ORDER BY schema_name;

-- Q2: All user tables
SELECT table_schema, table_name
FROM information_schema.tables
WHERE table_type = 'BASE TABLE'
  AND table_schema NOT IN ('information_schema', 'pg_catalog', 'pg_toast')
  AND table_schema NOT LIKE 'pg_temp_%'
ORDER BY table_schema, table_name;

-- Q3: All columns
SELECT table_schema, table_name, column_name, data_type,
       (is_nullable = 'YES') AS is_nullable
FROM information_schema.columns
WHERE table_schema NOT IN ('information_schema', 'pg_catalog', 'pg_toast')
  AND table_schema NOT LIKE 'pg_temp_%'
ORDER BY table_schema, table_name, ordinal_position;

-- Q4: All primary key columns
SELECT tc.table_schema, tc.table_name, kcu.column_name
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
  AND tc.constraint_schema = kcu.constraint_schema
WHERE tc.constraint_type = 'PRIMARY KEY'
  AND tc.table_schema NOT IN ('information_schema', 'pg_catalog', 'pg_toast')
  AND tc.table_schema NOT LIKE 'pg_temp_%';

-- Q5: All foreign key relationships
SELECT
    tc.table_schema,
    tc.table_name,
    kcu.column_name,
    ccu.table_schema AS referenced_schema,
    ccu.table_name  AS referenced_table,
    ccu.column_name AS referenced_column,
    tc.constraint_name
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
  AND tc.constraint_schema = kcu.constraint_schema
JOIN information_schema.referential_constraints rc
  ON tc.constraint_name = rc.constraint_name
  AND tc.constraint_schema = rc.constraint_schema
JOIN information_schema.key_column_usage ccu
  ON rc.unique_constraint_name = ccu.constraint_name
  AND rc.unique_constraint_schema = ccu.constraint_schema
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_schema NOT IN ('information_schema', 'pg_catalog', 'pg_toast')
  AND tc.table_schema NOT LIKE 'pg_temp_%'
ORDER BY tc.table_schema, tc.table_name, kcu.column_name;
```

**Assembly logic** (Rust pseudocode):
```
1. Run Q1 → schema_names: Vec<String>
2. Run Q2 → table_rows: Vec<(schema, table)>
3. Run Q3 → column_rows: Vec<(schema, table, column, data_type, is_nullable)>
4. Run Q4 → pk_rows: Vec<(schema, table, column)>
5. Run Q5 → fk_rows: Vec<(schema, table, column, ref_schema, ref_table, ref_col, constraint_name)>

6. Build pk_set: HashMap<(schema, table), HashSet<column>>
7. Build fk_map: HashMap<(schema, table, column), ForeignKeyRef>
8. Build fk_list_map: HashMap<(schema, table), Vec<ForeignKey>>
9. Build column_map: HashMap<(schema, table), Vec<PgColumn>> using column_rows + pk_set + fk_map
10. Build table_map: HashMap<schema, Vec<PgTable>> using table_rows + column_map + fk_list_map
11. Assemble SchemaTree using schema_names + table_map
12. Emit schema_progress events during step 10 (per table assembled)
```

Use `sqlx::query!` macro where compile-time checking is possible, or `sqlx::query_as` with anonymous row types. Since the queries have boolean casts (`is_nullable = 'YES'`), prefer `sqlx::query!` with explicit type annotations.

**sqlx fetch pattern**:
```rust
let rows = sqlx::query!("SELECT schema_name FROM information_schema.schemata WHERE ...")
    .fetch_all(pool)
    .await
    .map_err(|e| AppError::ExtractionFailed(e.to_string()))?;
```

### AppState Restructure

[Source: architecture/11-backend-architecture.md#114-tauri-managed-state]

Current `AppState` (after Story 2.1):
```rust
pub struct AppState {
    pub connection_manager: ConnectionManager,  // NOT wrapped in Mutex
}
```

After this story:
```rust
pub struct AppState {
    pub connection_manager: tokio::sync::Mutex<ConnectionManager>,
    pub schema_extractor: services::schema_extractor::SchemaExtractor,
}
```

The `connection_manager` must be wrapped in `tokio::sync::Mutex` (not `std::sync::Mutex`) because `connect()` and `disconnect()` are async methods. `std::sync::Mutex` cannot be held across `.await` points.

**Impact on existing `test_connection` command** — the handler in `commands/connection.rs` must change from:
```rust

```
to:
```rust
state.connection_manager.lock().await.test(&params).await
```

[Source: architecture/17-coding-standards.md]

- **IPC Gateway**: All `invoke()` calls go through `src/commands/index.ts`. Schema components read from Zustand only — they never call `invoke` directly.
- **Field selectors**: Every `useAppStore(...)` call must pass a selector. No component calls `useAppStore()` without one.
- **Command handlers thin**: `connect_and_extract_schema` delegates entirely to `ConnectionManager.connect()` + `SchemaExtractor.extract()`. Handler body ≤ 15 lines.
- **Rust error propagation**: All fallible calls use `?`. No `.unwrap()` in non-test code.
- **Drop password after use**: After `connect()` returns, `password` goes out of scope (it's passed by value). No explicit `drop()` needed.
