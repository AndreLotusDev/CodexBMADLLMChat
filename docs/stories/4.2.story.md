# Story 4.2: Persist Annotations Across Sessions

## Status

Done

## Story

**As a** user,
**I want** my table and column annotations to be saved automatically,
**so that** I don't have to re-annotate my schema every time I open the app.

## Acceptance Criteria

1. Annotations are saved automatically when the user types (no explicit save button required)
2. Annotations are stored locally in SQLite, keyed by connection profile + schema + table + column
3. When reconnecting with a saved profile, existing annotations are restored and displayed in the tree
4. Deleting a connection profile also deletes all associated annotations
5. Annotations survive app restarts

  - [x] `upsert` SQL ŌĆö architecture ┬¦11.2 prescribes the `INSERT ŌĆ” ON CONFLICT(connection_profile_id, schema_name, table_name, column_name) DO UPDATE` pattern. Generate a fresh UUID for the `id` column on INSERT; ON CONFLICT path keeps the existing row's `id`. After the write, re-read the row by its natural key so the returned `Annotation` carries the canonical `id` (whether new or pre-existing) and `updated_at`:
    ```rust
    pub fn upsert(&self, params: &UpsertAnnotationParams) -> Result<Annotation, AppError> {
        if params.text.chars().count() > 500 {
            return Err(AppError::AnnotationTooLong);
        }
        let new_id = uuid::Uuid::new_v4().to_string();
        let updated_at = chrono::Utc::now().to_rfc3339();
        let guard = self.conn.lock().map_err(|e| AppError::Internal(format!("DB lock poisoned: {e}")))?;
        let result = guard.execute(
            "INSERT INTO annotations
                 (id, connection_profile_id, schema_name, table_name, column_name, text, updated_at)
             VALUES (?1, ?2, ?3, ?4, ?5, ?6, ?7)
             ON CONFLICT(connection_profile_id, schema_name, table_name, column_name)
             DO UPDATE SET text = excluded.text, updated_at = excluded.updated_at",
            params![
                new_id,
                params.profile_id,
                params.schema_name,
                params.table_name,
                params.column_name,
                params.text,
                updated_at,
            ],
        );
        // FK violation on connection_profile_id surfaces as a constraint error ŌĆö route to ProfileNotFound.
        // The text_max_length CHECK is also a constraint error, but we already pre-validated above.
        match result {
            Ok(_) => {}
            Err(rusqlite::Error::SqliteFailure(sf, _))
                if sf.code == rusqlite::ErrorCode::ConstraintViolation =>
            {
                return Err(AppError::ProfileNotFound);
            }
            Err(e) => return Err(AppError::Internal(e.to_string())),
        }
        // Re-read by natural key so we return the canonical row (id may differ from new_id on UPDATE path).
        guard.query_row(
            "SELECT id, connection_profile_id, schema_name, table_name, column_name, text, updated_at
             FROM annotations
             WHERE connection_profile_id = ?1
               AND schema_name = ?2
               AND table_name  = ?3
               AND ((column_name IS NULL AND ?4 IS NULL) OR column_name = ?4)",
            params![params.profile_id, params.schema_name, params.table_name, params.column_name],
            |row| Ok(Annotation {
                id: row.get(0)?,
                connection_profile_id: row.get(1)?,
                schema_name: row.get(2)?,
                table_name: row.get(3)?,
                column_name: row.get(4)?,
                text: row.get(5)?,
                updated_at: row.get(6)?,
            }),
        ).map_err(|e| AppError::Internal(e.to_string()))
    }
    ```
  - [x] The `column_name IS NULL` branch is required because SQL `column_name = NULL` is always false. The `uq_annotation` UNIQUE constraint treats `NULL` slots as distinct, but the ON CONFLICT clause handles them correctly via the named conflict target. The re-read query above is the only place we have to match NULL explicitly.

  - [x] Standard debounce hook returning a stable callback that fires after `delay` ms of quiescence ŌĆö used by `AnnotationInput` (Task 12). API:
    ```typescript
    import { useEffect, useRef, useCallback } from 'react'

    // Returns a stable wrapped callback that fires `delay` ms after the last call,
    // and a `flush()` that fires immediately (used when the component unmounts mid-debounce).
    export function useDebounce<TArgs extends unknown[]>(
      fn: (...args: TArgs) => void | Promise<void>,
      delay: number,
    ): { run: (...args: TArgs) => void; flush: () => void; cancel: () => void } {
      const fnRef = useRef(fn)
      const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null)
      const lastArgsRef = useRef<TArgs | null>(null)

      // Keep latest fn without breaking memoization of the returned callbacks.
      useEffect(() => { fnRef.current = fn })

      const cancel = useCallback(() => {
        if (timerRef.current !== null) { clearTimeout(timerRef.current); timerRef.current = null }
      }, [])

      const flush = useCallback(() => {
        if (timerRef.current !== null && lastArgsRef.current !== null) {
          clearTimeout(timerRef.current)
          timerRef.current = null
          void fnRef.current(...lastArgsRef.current)
        }
      }, [])

      const run = useCallback((...args: TArgs) => {
        lastArgsRef.current = args
        cancel()
        timerRef.current = setTimeout(() => {
          timerRef.current = null
          void fnRef.current(...args)
        }, delay)
      }, [cancel, delay])

      // Cleanup on unmount: cancel pending timer.
      useEffect(() => () => cancel(), [cancel])

      return { run, flush, cancel }
    }
    ```

interface Annotation {
  id: string;                  // UUID v4, assigned by Rust on INSERT
  connectionProfileId: string; // FK to connection_profiles.id; cascade-delete

  tableName: string;

### SQLite Schema (already in `migrations/001_initial.sql`)

```sql
CREATE TABLE IF NOT EXISTS annotations (
    id                    TEXT PRIMARY KEY,
    connection_profile_id TEXT NOT NULL,
    schema_name           TEXT NOT NULL,
    table_name            TEXT NOT NULL,
    column_name           TEXT,                              -- NULL = table-level
    text                  TEXT NOT NULL,
    updated_at            TEXT NOT NULL DEFAULT (datetime('now')),

    CONSTRAINT fk_profile
        FOREIGN KEY (connection_profile_id)
        REFERENCES connection_profiles(id)
        ON DELETE CASCADE,

    CONSTRAINT text_max_length CHECK (length(text) <= 500),

    CONSTRAINT uq_annotation
        UNIQUE (connection_profile_id, schema_name, table_name, column_name)
);

CREATE INDEX IF NOT EXISTS idx_annotations_profile
    ON annotations(connection_profile_id);
CREATE INDEX IF NOT EXISTS idx_annotations_lookup
    ON annotations(connection_profile_id, schema_name, table_name);
```

Notes on the schema that matter for this story's implementation:
- **`column_name` is nullable** ŌĆö table-level annotations have `column_name IS NULL`. SQL `column_name = NULL` is never true, so the natural-key re-read in `upsert` MUST use `(column_name IS NULL AND ? IS NULL) OR column_name = ?` (Task 3).
- **`uq_annotation` UNIQUE** is the conflict target for `INSERT ŌĆ” ON CONFLICT ŌĆ” DO UPDATE`. SQLite treats NULL slots as distinct in vanilla UNIQUE constraints, but the named conflict target with the right column ordering still resolves table-level upserts correctly.
- **`idx_annotations_lookup`** is what makes `list_for_profile` (and the natural-key re-read) fast at MVP scale.
- **`text_max_length` CHECK** is a secondary defense ŌĆö the primary defense is the explicit 501-char check at the top of `upsert` (Task 3), which surfaces as `AnnotationTooLong` rather than the generic constraint violation.

[Source: architecture/9-database-schema.md#91-sqlite-schema-ddl, src-tauri/src/migrations/001_initial.sql:19-43]
