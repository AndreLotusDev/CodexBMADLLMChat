# Story 3.2: Table & Column Annotation

## Status

Done

## Story

**As a** user,
**I want** to add plain-text descriptions to tables and columns,
**so that** the generated LLM prompt includes context that isn't in the schema alone.

## Acceptance Criteria

1. Clicking a table or column opens an inline annotation input (sidebar or popover)
2. Annotations accept plain text up to 500 characters
3. Annotated tables/columns are visually marked in the tree (e.g., a small icon)
4. Annotations are included in the generated prompt alongside the DDL
5. Clearing an annotation removes the visual marker

## Dev Notes

### Scope: In-Memory Only — Persistence Is Epic 4.2

[Source: docs/prd/epic-4-persistence-profiles-annotations.md#story-42, architecture/4-data-models.md#43-annotation]

The PRD splits annotations across two epics:

- **Epic 3.2 (this story)**: UI for adding/clearing annotations + visual marker + data structure ready for prompt generation (Story 3.3).
- **Epic 4.2 ("Persist Annotations Across Sessions")**: SQLite round-trip (`upsert_annotation` / `load_annotations` / `delete_annotation` IPC commands), keyed by `connectionProfileId`. Requires Epic 4.1 ("Save & Load Connection Profiles") to land first because annotations have a FK to `connection_profiles.id`.

**Codebase state confirms this ordering is correct as of HEAD:**

- `Cargo.toml` declares only `sqlx` (PostgreSQL). No `rusqlite`, no `keyring` (WinCred).
- `src-tauri/src/db.rs` is a TODO stub.
- `src-tauri/src/migrations/` is empty (no `001_initial.sql`).
- `src-tauri/src/repositories/mod.rs` is a stub comment — no `annotation_repository.rs`, no `profile_repository.rs`.
- `src/commands/index.ts` exports only connection + disconnect commands. No `loadAnnotations` / `upsertAnnotation` / `deleteAnnotation`.
- `src/types/index.ts` has no `Annotation` interface and no `ConnectionProfile` type.
- The Zustand store has no `activeProfile` concept yet — connections in the current build are ad-hoc per session (Story 2.1 stored them outside of a saved profile).

**Implication for Story 3.2:** annotations live entirely in Zustand for this story. The user opens a connection, adds annotations, sees the visual markers, and the data is available for the upcoming Story 3.3 prompt generator. On disconnect / app close, annotations are lost — that's the persistence gap Epic 4.2 fills.

**Forward-compatibility design choices:**

- Keep the architecture §4.3 `Annotation` type shape verbatim — `id` populated via `crypto.randomUUID()`, `connectionProfileId` populated with the empty-string sentinel `''`. When Epic 4.2 lands, the `connectionProfileId` field will be populated from the active profile and the upsert/load IPC commands will be wired; the type itself does not change.
- Key format from `buildAnnotationKey()` is the canonical `${schema}.${table}.${col ?? ''}` — exactly what the SQLite unique constraint `UNIQUE (connection_profile_id, schema_name, table_name, column_name)` will key on (architecture §9.1). The trailing dot for table-level keys is preserved.
- No `invoke()` calls in this story — `src/commands/index.ts` is untouched.

### AC #4 — "Included in the Generated Prompt"

[Source: docs/prd/epic-3-table-selection-prompt-generation.md#story-33]

AC #4 says annotations are included in the generated prompt. The prompt-generation feature itself is the next story (3.3) — it does not exist at HEAD. AC #4 is therefore satisfied jointly:

- **This story (3.2)** produces the data layer: `useAppStore.getState().annotations` is a `Map<string, Annotation>` keyed by `buildAnnotationKey()`, with every entry's shape matching architecture §4.3.
- **Story 3.3** will read that Map when assembling the prompt, formatting annotations as SQL comments per the AC: "DDL for each selected table, and any annotations formatted as comments".

The expected consumption pattern in Story 3.3 (architecture §6.3 Prompt Generator) for each selected table key `${schema}.${table}`:

1. Look up the table-level annotation with `annotations.get(buildAnnotationKey(schema, table, null))`.
2. For each selected column, look up the column-level annotation with `annotations.get(buildAnnotationKey(schema, table, columnName))`.

This story is therefore "AC #4 complete from the data side; visible-in-prompt closes when 3.3 lands". No prompt rendering is built here.

### Annotation Data Model

[Source: architecture/4-data-models.md#43-annotation, architecture/10-frontend-architecture.md#102-state-management-architecture]

```typescript
interface Annotation {
  id: string                       // UUID — crypto.randomUUID() for in-memory
  connectionProfileId: string      // '' sentinel this story; Epic 4.2 populates from active profile
  schemaName: string
  tableName: string
  columnName: string | null        // null = table-level annotation
  text: string                     // max 500 chars (NFR — also enforced by DOM maxLength)
  updatedAt: string                // ISO 8601
}

type AnnotationKey = `${string}.${string}.${string | ''}`
// Examples: "public.users." (table) or "public.users.email" (column)
```

Architecture §10.2 prescribes the store layout exactly:

```typescript
annotations: Map<string, Annotation>
setAnnotation: (key: string, annotation: Annotation) => void
removeAnnotation: (key: string) => void
```

Use those names verbatim. No nested wrapper (e.g., do not introduce a `state.annotation.{}` substate) — the architecture's flat field is the contract.

### Annotation Key Convention — Critical Rule

[Source: architecture/17-coding-standards.md#171-critical-fullstack-rules]

§17.1 includes: **"Annotation Key Convention: Always use `buildAnnotationKey()` from `src/lib/utils.ts` — never construct keys inline."**

This is enforced this story (the util is added in Task 2) and binds every consumer onward. Story 3.1's selection key (`${schema}.${table}` / `${schema}.${table}.${col}`) is a separate format — do NOT conflate them. Selection keys have no trailing dot for table; annotation keys DO have a trailing dot for table-level (so a column key and a table key are never lexically equal — important for downstream prefix queries in Epic 4.2's SQLite layer).

| Concept | Key format | Built by | Source |
|---------|-----------|----------|--------|
| Selection (table) | `"public.users"` | inline `${schema}.${table}` | Story 3.1 |
| Selection (column) | `"public.users.email"` | inline `${schema}.${table}.${col}` | Story 3.1 |
| Annotation (table) | `"public.users."` (trailing dot) | `buildAnnotationKey(schema, table, null)` | This story |
| Annotation (column) | `"public.users.email"` | `buildAnnotationKey(schema, table, col)` | This story |

Annotation column key and selection column key happen to be identical in spelling — that is intentional and safe because they live in different containers (`annotations: Map<...>` vs `selectedColumns: Set<...>`).
